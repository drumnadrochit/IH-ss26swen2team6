1. Project Overview

Tour Planner is a full-stack web application for planning and logging bike, hike, running, and vacation tours. Users self-register, create tours, and record tour logs with statistics. Route data (distance, duration, map polyline) is fetched live from the OpenRouteService.org REST API and displayed on an interactive Leaflet map. The application runs in a three-container Docker Compose setup (PostgreSQL + ASP.NET Core backend + Nginx/React frontend).

This final submission completes the business logic and data access layer and adds everything that was missing or stubbed at the intermediate stage: full-text search across tour and log data (including computed attributes), import/export of tour data, externally-stored tour images, a working child-friendliness classifier, a Strategy-based fallback for route estimation, a Unit of Work for atomic multi-entity writes, hardened input validation, removal of committed secrets, a unique weather feature, and — most importantly — an explicit MVVM layer on the frontend.

4. MVVM Architecture in React

React has no built-in ViewModel concept, so the project defines its own minimal MVVM layer instead of relying on Zustand stores or component-local useState as the source of truth for business state.

MVVM RoleImplementation
Modelapi/*.ts (HTTP access), DTO types/*.ts, and store/authStore.ts (persisted session) — the actual data and server access.
ViewModelviewmodels/ViewModel.ts (base class), AuthViewModel, TourListViewModel, TourDetailViewModel — observable properties + commands, no JSX.
ViewPage/components under pages/ and components/ — render-only, read ViewModel properties, call ViewModel commands on user interaction.

ViewModel is a plain class holding a listener set and a version counter. Every mutating method ends with this.notify(), which increments version and notifies subscribers. The useViewModel(factory) hook creates one ViewModel instance per component (via useState) and subscribes to it with React's useSyncExternalStore, so the View re-renders whenever the ViewModel's state changes — the same effect two-way data binding gives you in Angular/WPF, without requiring a framework that supports it natively.

// View (TourDetailPage.tsx) - binds to the ViewModel, no business logic const vm = useViewModel(() => new TourDetailViewModel(id)); useEffect(() => { void vm.load(); }, [vm]); ... <h1>{vm.tour.name}</h1> <button onClick={() => vm.requestDeleteTour()}>Delete</button> // ViewModel (TourDetailViewModel.ts) - observable state + commands, no JSX async confirmDeleteTour(): Promise<void> { await deleteTour(this.tourId); // Model }

Transient, uncommitted form input (e.g. text typed into the Tour/TourLog create forms before Save is pressed) is kept as local component state, consistent with how MVVM implementations on other platforms treat input buffering — the moment the user submits, the ViewModel command is invoked and the result becomes part of the ViewModel's observable state.

6. Technical Steps, Failures & Solutions (Final Phase)

IssueRoot CauseSolution
Full-text search over computed attributes (popularity, child-friendliness) couldn't be expressed as a single SQL query Popularity and child-friendliness are computed in C# from a tour's logs, not stored columns Search is implemented in TourService.SearchToursAsync: load the user's tours (already Includes logs), map each to its TourResponse (which carries the computed fields), then filter in-memory across raw and computed fields. Acceptable given the per-user dataset size.
Adding lat/lon to Tour required a new EF migration without a live dev database to hand Coordinates from ORS geocoding were previously discarded after computing distance/duration Added nullable FromLat/FromLon/ToLat/ToLon columns and generated AddTourCoordinates via dotnet ef migrations add; applied automatically by the existing db.Database.Migrate() call on startup.
Directions API occasionally geocodes both endpoints but fails the subsequent directions call, leaving Distance/EstimatedTime at 0 ORS directions endpoint has its own failure modes independent of geocoding Added a haversine-distance fallback combined with the ITransportSpeedStrategy average speed per transport type, so a tour still gets a sensible (if approximate) distance/time instead of zero.
Uploaded images need to be served back to the browser without exposing the whole filesystem Images are intentionally stored outside the database/wwwroot, in a configurable directory Mapped a dedicated StaticFileMiddleware with a PhysicalFileProvider rooted at Storage:BasePath to the /uploads request path; both the Vite dev proxy and the production Nginx config forward /uploads to the backend.
React has no first-class ViewModel/observable primitive Hooks and Zustand encourage colocating business logic with components (Flux-style), which the intermediate review correctly flagged as not MVVM Implemented a ~25-line ViewModel base class around useSyncExternalStore (see Section 4) instead of pulling in a heavier MVVM/MobX framework, keeping the dependency footprint small.

7.1 Unique Feature: Current Weather at the Tour's Start Location

GET /api/tours/{'{id}'}/weather looks up the tour's geocoded FromLat/FromLon (captured when the tour's route is first resolved against OpenRouteService) and forwards them to Open-Meteo.com, a free weather API that requires no API key or registration. The Tour Detail page renders the result — temperature, wind speed, a human-readable description, and day/night — in a small WeatherWidget component. This was chosen as the mandatory unique feature because it is directly useful for trip planning (e.g. deciding whether today is a good day for a 60 km bike tour) and needed no additional backend infrastructure beyond one more Adapter (IWeatherClient) following the same shape as the existing OpenRouteService integration.

10. Unit Tests – Selection & Justification

The test suite contains 54 NUnit tests, all using Moq to isolate the class under test from its dependencies (repositories, HTTP clients, Unit of Work). Run with dotnet test inside backend/; all 54 pass.

Test ClassWhat is testedWhy critical
AuthServiceTests Register success/duplicate email/duplicate username/weak password; login success/wrong password/unknown email Authentication is the security boundary of the whole app — bugs here allow account takeover or data leakage.
TourServiceTests CRUD ownership checks; full-text search across name/log-comments/computed child-friendliness; image-set ownership check; coordinate lookup for weather; haversine fallback when ORS directions fail Contains the core business logic, including the per-user ownership guard that prevents cross-account data access, and the search behaviour the spec explicitly requires to cover computed attributes.
TourLogServiceTests Create/validate difficulty & rating ranges; ownership-scoped get/delete Logs drive the computed popularity/child-friendliness values — invalid values would silently corrupt them.
TourImportExportServiceTests Export mapping; import adds tours+logs in one Unit-of-Work call; rows with missing required fields are skipped; unknown transport type defaults safely; no-op when nothing valid to import Import is the one place the app writes multiple related entities from untrusted (file-based) input — must not partially commit or crash on malformed data.
ChildFriendlinessClassifierTests Unknown (no logs), Yes/Moderate/No thresholds This was a hardcoded stub at the intermediate stage; tests pin down the now-real behaviour.
TransportSpeedResolverTests Correct speed per TransportType; fallback when no strategy registered Feeds the haversine fallback estimate — a wrong speed silently produces a wrong ETA with no error.
GeoUtilsTests Haversine distance for identical points and a known city pair Used directly in the route-estimation fallback; an off-by-factor bug (e.g. radians vs degrees) would be invisible without a known-distance assertion.
FileImageStorageTests Valid JPEG saved and path returned; unsupported MIME type rejected; oversized file rejected; missing content-type rejected The only validation standing between an authenticated user and arbitrary file writes on the server's filesystem.
WeatherServiceTests Successful mapping; unknown weather code; upstream client returning null Wraps an external, unauthenticated third-party API — must degrade to null rather than throwing.
Test strategy: every service is tested against mocked interfaces (repositories, Unit of Work, HTTP clients) rather than a real database or network call, so the suite runs in well under a second and a failure always points at the service logic itself, not infrastructure.
